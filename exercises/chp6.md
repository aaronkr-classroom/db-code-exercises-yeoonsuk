## 연습문제

1. 디음 중 개체 심화 요소에 대한 설명으로 적절하지 않는 것은?
  1. 키 개체는 다른 개체의 부모 개체가 되므로 가장 먼저 추출해야 한다.
  2. 관계 개체는 ER 모텔에서 대개 관계에 해당한다. <----------------------------
  3. 코드 개체는 정보를 간단히 표현하기 위한 기호를 나타내는 개체다.
  4. 약한 개체는 소유 개체가 있어야 존재할수 있는 개체다.

2. 키 개체와 메인 개체의 차이점을 설명하시오.
키 개체: 부모 개체가 존재하지 않는 독립적인 핵심 개체입니다. (예: 고객, 사원)

메인 개체: 키 개체들 간의 결합이나 업무적 관련성으로 인해 하위로 생성되는 개체입니다. (예: 주문, 계약)

3. 관계 개체와 메인 개체의 차이점을 설명하시오
메인 개체: 비즈니스의 주요 트랜잭션을 나타냅니다.

관계 개체: 두 개체 간의 다대다(M:N) 관계를 1:M 관계로 해소하기 위해 논리적 모델링 과정에서 인위적으로 만들어진 개체입니다.

4. 다음과 같은 병렬 관계를 직렬 관계로 변환하여 ER 다이어그램으로 표현하시오. (슬라이드 나 책 235쪽)
병렬 관계 -> 직렬 관계 총 4 단계
1. 새로운 관계 개체 추가
강사와 강좌 사이에 이들의 관계 이력을 관리할 새로운 개체를 만듭니다. 개체 이름은 활동내역(또는 강좌담당) 정도로 정할 수 있습니다.

2. 관계 속성 통합 및 구분자 추가
기존의 개발하다, 강의하다, 평가하다 라는 세 개의 관계를 하나로 통합합니다. 대신 이것이 어떤 활동인지 구분하기 위해 새로운 활동내역 개체에 활동구분 이라는 속성을 새로 만듭니다. (이 속성 값으로 개발, 강의, 평가 라는 데이터가 들어갑니다)
각 관계에 매달려 있던 개발일자, 강의일자, 평가일자를 통합하여, 새로운 개체에 일자 라는 단일 속성으로 묶어줍니다.

3. 식별자 상속

새로 만들어진 활동내역 개체는 누구의 어떤 강좌에 대한 내역인지 알아야 하므로 부모 개체들로부터 기본 키를 물려받습니다.

강사 개체의 id와 강좌 개체의 번호를 외래 키로 가져옵니다.

4. 1:N 직렬 관계로 재설정

기존의 병렬 선들을 모두 제거하고 새로 만든 활동내역 개체를 중심으로 양옆을 1:N 관계로 이어줍니다.

[강사] 1 ---- N [활동내역]

[강좌] 1 ---- N [활동내역] ---

변환된 최종 데이터 구조 요약
결과적으로 ER 다이어그램은 [강사] - [활동내역] - [강좌]가 일렬로 늘어선 직렬 형태가 되며 각 개체가 가지는 속성은 다음과 같이 정리됩니다.

강사 개체: id (PK), 이름, 휴대폰, 전공

활동내역 개체 (신규): 강사_id (FK), 강좌_번호 (FK), 활동구분(개발/강의/평가 중 택 1), 일자

강좌 개체: 번호 (PK), 강좌명, 내용, 수강대상

6. (________) 안에 적합한 단어는?
  > 원래 존재하지는 않았지만 필요에 따라 설계자가 추가한 속성을 (설계)속성이라고한다.

6. 슈퍼 서브 타업을 정제할 때 어떤 경우에 슈퍼타입 개체를 기준으로 통합하는지 설명하시오.
설명: 서브타입 개체들이 가지는 고유 속성의 개수가 매우 적거나, 트랜잭션이 서브타입을 개별적으로 구별하지 않고 전체 데이터를 한 번에 처리하는 경우가 많을 때 하나의 통합된 테이블로 합칩니다.

7. 일반회원과 VIP회원을 처리하는 모든응용 프로그램이 병확히 구분되고， 추후 추가될 응용프로그램도 마찬가지다. 아래 슈퍼-서브 타입을 바르게 정제한 결과를 ER 다이어그램으로 표현하시오.
기존의 삼각형(슈퍼-서브 타입 기호)과 [회원] 개체가 사라지고 다음과 같이 완전히 독립된 2개의 개체로 나뉘어 그려집니다.

1. [일반회원] 개체
슈퍼타입에서 내려온 공통 속성과 본래 가지고 있던 고유 속성을 모두 합쳐서 가집니다.

식별자(PK): 번호 (검은색 타원)

일반 속성(타원): 이름, 휴대폰, 누적구매액, 구매횟수, 할인쿠폰수

2. [VIP회원] 개체
마찬가지로 슈퍼타입의 공통 속성과 VIP만의 고유 속성을 모두 가집니다.

식별자(PK): 번호 (검은색 타원)

일반 속성(타원): 이름, 휴대폰, 누적구매액, 세부등급, 할인율, 세일리지

8. 사원 가운데 일부는 관리자이므로 유일한 ‘관리자변호’와 ‘직책’， ‘임명일자’， ‘임명기간’ 속성이 필요하여 아래와 같이 사원 개체를 표현했는데， 사원 가운데 관리자의 비율은 5% 미만이다. 이 ER 델을 좀 더 효율적으로 정제해서 ER 다이어그램으로 표현하시오.
1. 개체 분리 및 속성 배분
하나의 [사원] 개체를 [사원](기본 정보)과 [관리자](관리자 전용 정보) 두 개의 개체로 쪼갭니다.

[사원] 개체 (메인 개체)

식별자(PK): 사번 (검은색 타원)

일반 속성: 이름, 휴대폰, 입사년도, 연봉

(팁: 여기에 관리자여부(Y/N) 같은 구분자 속성을 하나 추가해주면 검색 효율이 더 좋아집니다.)

[관리자] 개체 (서브 개체)

식별자(PK/FK): 사번 (사원 개체에서 물려받아 검은색 타원으로 표시 누구의 관리자 정보인지 알아야 하므로 외래키이자 기본키 역할을 동시에 합니다.)

일반 속성: 관리자번호, 직책, 임명일자, 임명기간

2. 관계 설정 (1:1 관계)

두 개체 사이를 선으로 연결하고 마름모(관계)를 그려줍니다. 관계의 이름은 소속되다 또는 임명되다 등으로 할 수 있습니다.

모든 사원이 관리자는 아니므로, 사원(1) - 관리자(0 또는 1) 의 1:1 (선택적) 관계가 형성됩니다.

9. 다음과 같이 책을 분류하기 위한 도서 코드code 개체 타입을 ER 다이어그램으로 표현하시오.
  > 도서코드
  > 1: 인문， 2 공학， 3 미술， 4 음악， 5. 취미,...

이 개체는 매우 단순한 구조를 가집니다. 식별자(PK) 역할을 할 번호와 그 번호가 뜻하는 이름 딱 두 가지 속성만 있으면 됩니다.

1. 개체 (Entity)

[도서코드] (또는 도서분류)라는 이름의 직사각형을 그립니다.

2. 속성 (Attributes)
직사각형에서 두 개의 선을 뽑아 타원을 그리고 아래의 속성을 연결합니다.

식별자(PK): 코드번호 (1, 2, 3, 4, 5... 가 들어갈 자리입니다. 기본키이므로 타원을 검은색으로 칠하거나, 글자 아래에 밑줄을 그어 표시합니다.)

일반 속성: 분류명 (인문, 공학 등 실제 텍스트가 들어갈 자리입니다 일반 빈 타원으로 그립니다)

1. 가운데에 [도서코드]라고 적힌 네모 상자를 그립니다.

2. 상자 위쪽으로 선을 두 개 뽑아 동그라미(타원)를 두 개 그립니다.

3. 첫 번째 타원 안에는 코드번호라고 적고, 타원을 검게 칠합니다.

4. 두 번째 타원 안에는 분류명이라고 적고, 색칠하지 않은 빈 상태로 둡니다.

10. 다음 중 속성을 검증하는 방법으로 적절하지 않는 것은?
  1. 각각 의미가 다른 여러 개의 속성을 묶어서 하나의 속성으로 정의한 것이라면 다중치 속성으로 표현해야 한다. <------------------
  2. 모든 개체가 통일한 하나의 값만 갖는 속성이 있다면 제거해야 한다.
  3. 코드 값을 갖는 속성이 현업에서 통상적으로 시용되는 코드 값이 아니면 코드 사용을 삼가야 한다.
  4. 전산 처리에 필요한 플래그flag 형태의 속성은 제외해야 한다
